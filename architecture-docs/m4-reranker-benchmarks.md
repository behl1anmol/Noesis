# M4 Reranker — Gate Benchmarks

Measured results behind the M4 default-on/off decision (Finding 2, ADR-19).
Generated by `tests/eval/test_golden.py -m golden` on the 40-query golden set
(14 nl / 14 symbol / 12 structural). Candidate depth 50, `top_k` 10, RRF
fusion. **Quality metrics are hardware-independent and final; latency is
hardware- and precision-specific and is reported, not gated.**

## Compute environment

| Field | Value |
|---|---|
| Runtime | Google Colab, NVIDIA Tesla T4 (16 GB) |
| torch | 2.12.1+cu130, `cuda_available=True` |
| Precision | fp32 (sentence-transformers default; no fp16/tensor-core path) |
| Device (logged) | embedder `cuda`, reranker `cuda` |
| Embedder | nomic-ai/CodeRankEmbed (137M, 768-dim) |
| Reranker | BAAI/bge-reranker-v2-m3 (568M, XLM-RoBERTa-large cross-encoder) |

Device was recorded in-run (core/compute.py + `resolved_device`) after an
earlier run left the device ambiguous — a latency number without a recorded
device is not a measurement (dev lesson 4).

**Both Colab runs executed on the T4.** An earlier run's ~12 s latency was
first misattributed to CPU execution; it was not — the reranker was on the
GPU both times. The 12 s is the true fp32 T4 cost (see Interpretation). No
clean CPU-only latency was captured; the local i5-7200U run was cancelled
before completion. A CPU figure would only be *worse* and does not change the
decision (see below).

## Quality — M4 gate: hybrid+rerank vs same-run hybrid

Same-run comparison by design: both channels share the corpus, models and
labels, so no stale stored baseline can skew it (dev lesson 3).

| category | metric | hybrid | hybrid+rerank | delta |
|---|---|---|---|---|
| nl | recall@5 | 0.607 | 0.714 | **+0.107** |
| nl | recall@10 | 0.643 | 0.750 | **+0.107** |
| nl | ndcg@10 | 0.498 | 0.615 | **+0.117** |
| symbol | recall@5 | 0.857 | 0.929 | +0.071 |
| symbol | recall@10 | 0.857 | 0.929 | +0.071 |
| symbol | ndcg@10 | 0.636 | 0.732 | **+0.096** |
| structural | recall@5 | 0.875 | 0.958 | +0.083 |
| structural | recall@10 | 0.875 | 0.958 | +0.083 |
| structural | ndcg@10 | 0.743 | 0.848 | **+0.104** |
| **overall** | **recall@5** | 0.775 | 0.863 | **+0.088** |
| **overall** | **recall@10** | 0.787 | 0.875 | **+0.088** |
| **overall** | **ndcg@10** | 0.620 | 0.726 | **+0.106** |

**Verdict: reranking is a decisive, uniform quality win** — every metric, every
category improves, zero regressions. A prior run on a slightly earlier repo
snapshot corroborated it (overall NDCG@10 +0.099). This clears ADR-19's
"measurably wins on NDCG@10" bar.

## Latency (fp32, T4) — full search call per query

| channel | p50 (ms) | p95 (ms) |
|---|---|---|
| sparse | 7.2 | 8.2 |
| dense | 19.4 | 25.6 |
| hybrid | 60.2 | 67.4 |
| **hybrid+rerank** | **12180.1** | **13400.1** |

Reranking adds a ~200× latency multiplier over hybrid: ~12.2 s p50 / ~13.4 s
p95 per query.

## Interpretation — why 12 s is real, not a defect

Three independent checks agree the latency is genuine model inference:

1. **Device confirmed** `cuda` for both models (logged in-run).
2. **Tokenization ruled out** — the truncation pre-count tokenizes 50 pairs in
   ~215 ms (measured against the cached tokenizer); the predict-side truncated
   batch tokenize is ~88 ms. Neither approaches seconds.
3. **FLOP lower bound matches** — 568M params × 512 tokens ≈ 582 GFLOP/sequence.
   On a T4 at fp32 peak (~8.1 TFLOPS), at 25–50 % real efficiency that is
   144–287 ms/sequence → **7.2–14.4 s for 50 pairs**. Observed 12.2 s p50 sits
   squarely inside that band.

The cost is intrinsic: a large (568M) cross-encoder, 50 candidates per query,
fp32 with no tensor-core acceleration.

## Decision (ADR-19, Finding 2 discharged)

**Ship default-off / opt-in** — the current config default
(`reranker.enabled = false`). Rationale:

- Quality gate: **passed** decisively (+0.106 NDCG@10 overall).
- Latency: ~13.4 s p95 is ~27× over the proposed 500 ms p95 budget (open
  question 1) — and this is the *GPU* number; CPU is worse. Rerank-by-default
  would make interactive search unusable on the product's target hardware
  (developer laptops, Risk 9).
- Therefore the reranker ships as a **per-request opt-in** (`rerank: true`)
  for quality-critical searches, off by default. The measured quality win
  justifies keeping the feature; the latency forbids making it the default.

The decision is robust to the missing clean-CPU number and to precision: even
the best realistic GPU lever below does not reach the 500 ms budget.

## Unmeasured latency levers (future, if rerank-latency ever matters)

Documented for completeness; none change the default-off decision, all are
Could-haves:

| Lever | Expected effect | Note |
|---|---|---|
| fp16 on GPU (T4 tensor cores) | ~3–5× faster | Still ~2.5–4 s p95 → over budget |
| Candidate depth 50 → 25 | ~2× faster | Config `reranker.candidates`; recall cost unmeasured |
| Smaller reranker | model-dependent | e.g. bge-reranker-base; quality cost unmeasured |
| GPU deployment + fp16 | combined | Could justify default-on *for GPU installs only* |

uv run pytest tests/eval/ -m golden -s -q
