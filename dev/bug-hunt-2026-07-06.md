# Codebase Bug Hunt — 2026-07-06

Four parallel adversarial reviews over disjoint slices of `src/noesis/`
(indexing pipeline; state/storage; retrieval/MCP/app lifecycle; web/watcher/jobs),
followed by an independent verification pass: every claim below was re-traced
against the source (and empirically tested where possible) before inclusion.
All 23 findings survived verification. Findings are ranked by severity;
each carries a concrete failure scenario and a fix plan.

**Verified-clean areas** (attacked, no defect found): gitfast `-z` parsers
(NUL framing, rename tokens, prefix stripping), chunker byte-exact
reconstruction and point-ID uniqueness, ast-grep line conversion (probed:
correct), embedder/reranker worker queues (locking, drain ordering,
generation-based device swap), Jinja templates and app.js DOM writes (no XSS —
autoescape + `textContent`/`CSS.escape` throughout), telemetry fire-and-forget
contract, `languages.py`, nested-`.gitignore` last-match-wins ordering.

## Severity: HIGH — correctness / data loss / service degradation

### H1. Git fast-path serves stale content forever for files dirty at anchor time and later reverted
- **Where:** `src/noesis/core/gitfast.py:120-178` (candidate set = `diff anchor..HEAD ∪ status`) with `src/noesis/core/indexer.py:240-241` (anchor advance)
- **Defect:** A file with uncommitted edits during run N is indexed with its dirty hash; the anchor still advances to HEAD. If the file is then reverted to HEAD content (`git checkout --`, `git stash`, editor undo), run N+1 finds it in *neither* the diff nor `status` — so `hashdiff.partition` carries the stale stored hash forward as "unchanged" on every subsequent fast-path run. The index serves the reverted-away content indefinitely; a full hash-walk would catch it. Violates §3.2 rule 1 ("candidacy only shrinks hashing work, never decides changed/unchanged") and the M7 identical-partitioning exit criterion.
- **Failure scenario:** Edit `foo.py`, reindex (dirty content embedded), then `git checkout -- foo.py`. Every future auto/manual reindex keeps serving the old edit in search results until an unrelated commit touches `foo.py` or `git.fast_path=false`.
- **Fix plan:** Persist the status-derived dirty path set alongside the anchor when advancing it (new column or JSON in `projects`), and union it into the next run's candidates — a file dirty at run N is always a candidate at run N+1 (strictly widens; rule-1-safe). Simpler alternative: refuse to advance the anchor when `git status` is non-empty (costs full walks during active development).

### H2. Deleting a watched project permanently wedges the watcher flush pipeline
- **Where:** `src/noesis/core/watcher.py:222-237` (`_flush`), `watcher.py:138-148` (`set_watch`), trigger via `src/noesis/core/dashboard.py:416-439` (`delete_project`)
- **Defect:** `set_watch(project_id, False)` unschedules the OS watch but never purges the project's buffered events from `_accum`/`_queue`. After `delete_project` removes the `projects` row, the next `_flush` hits `FOREIGN KEY constraint failed` (`pending_changes.project_id REFERENCES projects(id)`, `PRAGMA foreign_keys=ON`). Because `_accum.clear()` sits at the *end* of `_flush`, the poisoned bucket is retried — and re-raises — on every 0.5 s tick forever.
- **Failure scenario:** Delete a project while it's being edited. Until process restart: (a) pending-change recording stops (or double-counts `watcher_stats` for projects flushed before the poisoned bucket each tick), and (b) `_maybe_auto_reindex` — which runs after `_flush` in the same `try` — never executes again, killing auto-reindex for **all** projects.
- **Fix plan:** In `set_watch(..., False)` also drop `_accum[project_id]`, `_seen_since_flush[project_id]`, `_last_event[project_id]`; additionally make `_flush` fault-isolate per project (per-bucket try/except, discard the bucket on `IntegrityError`) so one bad project can never wedge the loop.

### H3. Auto-reindex silently loses its re-trigger when a scoped launch is skipped
- **Where:** `src/noesis/core/watcher.py:239-268` (`_maybe_auto_reindex` / `_launch_scoped`) with `src/noesis/core/jobs.py:66-72` (already-running guard)
- **Defect:** `_maybe_auto_reindex` deletes `_last_event[project_id]` before launching. When `jobs.launch_index_run` skips because a run is already `running`, it returns the *existing* run's id with the same `status: "accepted"` shape — the skip is invisible to the watcher, which believes it fired. jobs.py's own docstring ("pending rows for a skipped scoped launch survive and re-trigger on the watcher's next quiet period") is false: nothing re-adds `_last_event` without a new filesystem event.
- **Failure scenario:** Save `foo.py` while a long manual reindex is running (its hash-diff already passed `foo.py`). Quiet period elapses → launch skipped → re-trigger state gone. With auto-reindex ON, the dashboard shows "1 pending" indefinitely and search serves stale code until the user edits another file or clicks reindex. Bonus: `auto_runs` is bumped and "auto-reindex launched" logged for the skipped launch, corrupting usage stats.
- **Fix plan:** Make `launch_index_run` distinguish a skip (e.g. `"status": "already_running"`); in `_launch_scoped`, on skip restore `_last_event[project_id] = time.monotonic()` so the quiet-period check retries, and only bump `auto_runs` on a real launch.

### H4. Hybrid search results and rerank candidate pool silently capped at 50 regardless of `top_k` / `reranker.candidates`
- **Where:** `src/noesis/core/vectorstore.py:269,323-345` (fixed `prefetch_limit=50`), `src/noesis/core/retriever.py:54-62` (never passes `prefetch_limit`)
- **Defect:** RRF fusion can only rank points returned by the two prefetches (50 dense + 50 sparse), but the prefetch limit is never raised to cover `top_k` (REST/MCP allow up to 100) or the configured rerank candidate depth. Independently found by two reviewers.
- **Failure scenario:** `top_k=100` hybrid search returns as few as ~50 hits with positions 51+ fillable only from the disjoint tails of two 50-deep lists. With `[reranker] candidates = 100`, a chunk ranked 55-60 on both channels can never enter the rerank pool — the documented "fused-candidate depth reranked per request" is silently not honored above ~50.
- **Fix plan:** In `retriever.search_code`, pass `prefetch_limit=max(top_k, candidates if apply_rerank else top_k)`; in `VectorStore.search`, default the hybrid prefetch limit to `max(prefetch_limit, top_k)`.

### H5. Shutdown closes SQLite before cancelled index tasks unwind — run rows stuck `running`, teardown blocks on the embed queue
- **Where:** `src/noesis/app.py:125-134` (`close_runtime_context`), reached from the lifespan (`app.py:158-160`) and `src/noesis/mcp/__main__.py:32`
- **Defect:** `task.cancel()` only *schedules* `CancelledError`; the function then synchronously closes the embedder worker and `ctx.conn` with no `await`, so the cancelled task resumes after the connection is closed. `execute_run`'s `except BaseException` handler (`indexer.py:254-258`) then calls `state.finish_run` on a closed DB → `sqlite3.ProgrammingError`; the run row stays `status='running'` until the next process's `fail_orphaned_runs`. Additionally `embedder.close()` drains queued work before its shutdown sentinel, so teardown can block on a full in-flight embed batch.
- **Failure scenario:** Stop the server mid-index (Ctrl-C / SIGTERM). Traceback on shutdown, run recorded as `running` despite the "cancellation marks the run failed" contract, and a slow shutdown while the embed batch finishes.
- **Fix plan:** Make teardown async at both call sites: after cancelling, `await asyncio.gather(*tasks, return_exceptions=True)`, then close embedder/reranker, then `conn.close()`.

### H6. Structural search ignores the project's persisted index scope (ADR-42)
- **Where:** `src/noesis/core/structural.py:127` (`discover_files(root_path)` with no config)
- **Defect:** Indexing builds its discovery config from the project row (`indexer.py:165-166`, `discovery_config_for_project`), but `_scan_sync` uses the default `DiscoveryConfig` — falsifying the module's guarantee that "structural search can never surface a file that indexing would have excluded."
- **Failure scenario:** Project registered with `extra_ignores=["vendored/**"]`: structural search returns matches from `vendored/` files that were never indexed. Project with `max_file_bytes=10MB` or `follow_symlinks=true`: structural search silently never scans files the index contains, returning falsely empty results.
- **Fix plan:** Resolve `discovery_config_for_project(conn, project_id)` in `structural_search` (project_id and conn are already available) and pass it to `discover_files`.

### H7. Any OSError while hashing a stored file is treated as deletion — its chunks are purged
- **Where:** `src/noesis/core/hashdiff.py:77-80` (`except OSError: continue`), consequence at `indexer.py` prune step
- **Defect:** "Vanished" and "unreadable" are conflated: a stored file that exists but momentarily fails to open (EACCES after a chmod, EIO/ESTALE on network filesystems) is omitted from `seen`, lands in `deleted`, and the indexer deletes its Qdrant points and state row. Contrast ADR-41: the same file failing during chunking is contained and retried; failing 30 lines earlier destroys index data.
- **Failure scenario:** NFS blip (EIO) during a reindex → file classified deleted → search silently loses it until a later run re-discovers it as new.
- **Fix plan:** Catch `FileNotFoundError`/`NotADirectoryError` as genuine deletion; for other OSErrors, carry the stored hash forward into `seen` (or record a per-file error) so chunks are never purged for a file that still exists.

## Severity: MEDIUM

### M1. Synchronous Qdrant query + client-side BM25 encoding run directly on the event loop
- **Where:** `src/noesis/core/retriever.py:54-62` (blocking `store.search` inside `async def search_code`)
- **Defect:** `QdrantClient.query_points` is a blocking HTTP round-trip, and hybrid/sparse channels additionally run fastembed BM25 query encoding client-side — all inline in a coroutine. Every search stalls the entire server (all MCP sessions, REST requests). Worst case: first hybrid search with a cold fastembed cache loads (or downloads) tokenizer assets synchronously, freezing the process for seconds-to-minutes.
- **Fix plan:** `hits = await asyncio.to_thread(store.search, ...)` in `search_code`.

### M2. Registering a project via the dashboard modal never displays it
- **Where:** `src/noesis/api/static/app.js:719-739` (`submit`), `app.js:111-113` (`applyProject` early-returns when no card exists)
- **Defect:** After a successful `POST /api/register` the modal closes and polling starts, but the poll path only mutates existing server-rendered cards; nothing creates a card or reloads. Totals update while the card list (or the "No projects indexed yet" empty state) stays stale until a manual refresh.
- **Fix plan:** On successful register, `location.reload()` (matches the server-rendered-first design, lesson 8), or render a card from the response payload.

### M3. REST maps a plain bad path to 409 Conflict (mixed-model contract) instead of 400
- **Where:** `src/noesis/api/routes.py:59-62` and `:85-88` (`except ValueError → 409`)
- **Defect:** `jobs.launch_index_run` raises `ValueError` both for a missing/non-directory `root_path` and (via `MixedModelError`, a `ValueError` subclass) the mixed-model guard. The dashboard adapter distinguishes them (`api/dashboard.py`, typed `MixedModelError` → 409, else 400); the REST surface doesn't, so an agent following the documented 409 recovery ("requires full re-index") acts on the wrong cause for a typo'd path.
- **Fix plan:** `status = 409 if isinstance(exc, MixedModelError) else 400` in both handlers.

### M4. Chunk metadata (`symbol_name`, `node_type`) wrong when a chunk starts with a comment
- **Where:** `src/noesis/core/chunker.py:80-93`
- **Defect:** `symbol_name` is read from `window[0]` while `node_type` comes from the first *named* piece; comments are named tree-sitter nodes, so a chunk opening with a leading comment (the chunker deliberately attaches comments to the following code) reports `node_type="comment"`, `symbol_name=None` even when it contains a function definition — degrading search payloads for a large fraction of real functions.
- **Fix plan:** Derive both fields from one piece: the first piece in the window with a non-None `symbol`, falling back to the first named non-comment piece, then `window[0]`.

### M5. `resolve_device` probes `torch.backends.mps.is_available()` unguarded — the exact crash `available_devices` defends against
- **Where:** `src/noesis/core/compute.py:39`
- **Defect:** `available_devices` (lines 55-66) wraps the same probe in try/except because "a torch build without the mps backend raises AttributeError (PR #10 review)"; `resolve_device` doesn't. On such a build with CUDA unavailable, every model load with `device=None` (i.e. every index run / dense search) raises `AttributeError` instead of falling back to CPU.
- **Fix plan:** Guard the probe (`try/except Exception → False`) or take the first entry of `available_devices()`.

### M6. Boolean config values coerced with `bool()` — the reranker kill switch fails open
- **Where:** `src/noesis/core/config.py:104,105,115`
- **Defect:** `bool(rrk.get("enabled", ...))` turns any non-empty value — notably the strings `"false"`, `"no"`, `"0"` — into `True`, while neighboring `int()`/`float()` fields fail loudly on the same class of typo. `enabled = "false"` silently loads the cross-encoder and turns reranking on; `preload = "no"` force-loads at startup; same for `git.fast_path`.
- **Fix plan:** Reject non-`bool` TOML values (`isinstance(v, bool)` or raise `ValueError`), matching the loud-failure behavior of the numeric fields.

### M7. `fail_orphaned_runs` in a second process kills the first process's live run and disarms the concurrent-run guard
- **Where:** `src/noesis/core/state.py:240-254`, called unconditionally from `build_runtime_context` (`app.py:77`)
- **Defect:** Startup marks *every* `running` row failed. In the documented two-process deployment (HTTP server + stdio MCP entry point sharing the DB, m6-agent-connection-guide), starting the second process mid-run mislabels the live run — and while mislabelled, `jobs.launch_index_run`'s already-running guard no longer fires, allowing two concurrent runs racing on the same Qdrant collection and `files` rows.
- **Fix plan:** Record an owner identity (PID + boot time, or instance UUID) on `index_runs` and only fail rows whose owner is dead; or gate the reset behind a flag the stdio entry point disables.

### M8. cwd-relative fastembed cache default breaks the offline/no-runtime-network guarantee
- **Where:** `src/noesis/prefetch.py:27` (`FASTEMBED_CACHE_DEFAULT = "data/fastembed_cache"`), consumed at `app.py:72`
- **Defect:** Prefetch and the running service only share a cache when launched from the same working directory. The stdio MCP server is typically spawned by an agent host with the *host's* cwd, so the first sparse/hybrid search misses the cache and re-downloads BM25 assets at runtime (network at query time, ADR-25-adjacent) — or fails outright offline.
- **Fix plan:** Anchor the default to an absolute location (derive from the resolved config's `db_path.parent` or a platform data dir) and use identical resolution in prefetch.py and app.py.

### M9. `follow_symlinks=True` walk has no cycle or duplicate guard
- **Where:** `src/noesis/core/discovery.py:165-167` (`os.walk(..., followlinks=True)`)
- **Defect:** `os.walk` does not detect symlink cycles; a self-referencing link (`loop -> .`) expands until `ENAMETOOLONG` kills each branch (discovery balloons, run effectively hangs), and a link into an already-walked subtree indexes every file twice under two relative paths (duplicate results, double embed cost).
- **Fix plan:** When following links, track visited directory identity (`st_dev, st_ino`) and prune revisits.

### M10. Usage-chart day bucketing breaks across DST transitions and mixes UTC/local keys
- **Where:** `src/noesis/api/static/app.js:370-394` (`dayKey`/`fillDays`) vs UTC `substr(ts,1,10)` day keys server-side (`core/dashboard.py:160-208`)
- **Defect:** The day axis steps back from local midnight in fixed 86,400,000 ms increments and re-keys in local time: across a DST boundary one calendar day appears twice and another is skipped; independently, local-time keys never match the server's UTC date buckets for users far from UTC (rows shift by a day).
- **Fix plan:** Iterate with `d.setDate(d.getDate() - 1)` (calendar-aware) and build keys in UTC (`getUTCFullYear()` etc.) to match the server.

## Severity: LOW

### L1. `db_path` not `expanduser()`-ed — `~/...` creates a literal `./~` directory
- **Where:** `src/noesis/core/config.py:95`; `state.connect` then `mkdir`s the literal path (`state.py:130`)
- **Defect/scenario:** `db_path = "~/noesis/state.sqlite"` silently puts the DB in a directory named `~` under the CWD; the user sees an empty service, and cleaning up the stray `~` dir is an `rm -rf ~` footgun.
- **Fix plan:** `.expanduser()` in `load_settings`.

### L2. Secret skip patterns `.aws/**` and `.ssh/**` only match at the project root
- **Where:** `src/noesis/core/discovery.py:64-65`
- **Defect/scenario:** Slash-containing gitignore patterns are anchored — verified empirically: `is_secret_path("sub/.aws/config") == False`. A vendored home-dir snapshot gets its nested `.aws/config` embedded and retrievable. Basename patterns (`credentials*`, `id_rsa*`) still catch the highest-value files.
- **Fix plan:** Use `**/.aws/**` and `**/.ssh/**` (or the directory forms `.aws/`, `.ssh/`).

### L3. Module-level `app = create_app()` import side effect
- **Where:** `src/noesis/app.py:194`, triggered by `mcp/__main__.py:17` importing helpers from `noesis.app`
- **Defect/scenario:** `python -m noesis.mcp` builds an entire unused FastAPI app (second FastMCP instance, static mount, config read from cwd) at import; any failure in HTTP-app construction (missing static dir, malformed config.toml) kills the stdio server before `main()` starts.
- **Fix plan:** Move `AppContext`/`build_runtime_context`/`close_runtime_context` to a side-effect-free module (e.g. `noesis/runtime.py`), or build the module-level app lazily (uvicorn `--factory`).

### L4. `register_project` check-then-insert is not atomic across processes
- **Where:** `src/noesis/core/state.py:149-172`
- **Defect/scenario:** Two processes registering the same root concurrently: the UNIQUE constraint preserves integrity but the loser gets a raw `sqlite3.IntegrityError` (→ 500 / MCP tool error) instead of the idempotent existing-id return. Single-process callers can't hit it.
- **Fix plan:** `INSERT ... ON CONFLICT(root_path) DO NOTHING` then re-SELECT (re-checking the model), or catch `IntegrityError` and retry the lookup.

### L5. No CSRF protection on body-less state-changing POSTs
- **Where:** `src/noesis/api/routes.py:79-88` (`POST /projects/{id}/reindex`), `api/dashboard.py` (`POST /api/projects/{id}/reindex-pending`)
- **Defect/scenario:** These POSTs take no JSON body, so a cross-site HTML form reaches them (Host is genuinely 127.0.0.1, passing TrustedHost, which only guards DNS rebinding). Exploitability is low — the attacker needs an unguessable 32-hex project id and the effect is a spurious reindex — but the class is open.
- **Fix plan:** Reject state-changing requests with a non-localhost `Origin`/`Referer` when present, or require a custom header from app.js so cross-origin form posts fail.

### L6. Chunk token budget counts non-whitespace *bytes*; the decided estimator counts *characters*
- **Where:** `src/noesis/core/chunker.py:104-116` (`_nws_prefix`/`_span_tokens`) vs the pinned `_token_estimate` (`:98-101`)
- **Defect/scenario:** CJK-heavy files (~3 bytes/char) hit the 800-"token" ceiling at ~⅓ of the intended content — chunks systematically ~3× smaller than the decided `ceil(non-ws chars / 4)` policy. Reconstruction and line-bound invariants still hold; calibration divergence only.
- **Fix plan:** Build the prefix array over characters (mapping char → UTF-8 length for offset bookkeeping), or explicitly re-decide the estimator as byte-based.

## Suggested fix order

1. **H1, H4, H6, H7** — retrieval/index correctness: wrong or missing results with no error signal.
2. **H2, H3, H5** — reliability of the watcher/job/shutdown machinery (wedges and stuck state).
3. **M1-M10** — quality-of-service and contract bugs; M1 (event-loop blocking) and M3 (status codes) are one-liners worth batching with the highs.
4. **L1-L6** — hardening; L2's pattern fix is a two-line change worth doing immediately despite the LOW rank.
